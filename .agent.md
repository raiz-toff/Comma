# Comma App Design Rules

The following design rules MUST be strictly followed for all new screens, UI components, and layout modifications. These rules ensure that the application maintains a cohesive, premium, and unified aesthetic across all tabs (Shifts, Analytics, Expenses, etc.).

## 1. True OLED Dark Theme
- **Root Backgrounds**: Always use pure OLED pitch black (`#000000`) for screen backgrounds, `SafeAreaView`s, and base containers.
- **Surface Cards**: Use a very deep slate gray (`#0d0d0d`) for individual cards, lists, Bento boxes, and elevated surfaces. Do NOT use lighter silvers like `#161615` for main content cards.
- **Surface Borders**: Use `#1f1f1f` for borders on `#0d0d0d` surfaces. The `borderWidth` should typically be `0.8`.
- **Button/Pill Surfaces**: Secondary elements like filter pills, date selectors, and arrow buttons can use `#161615` to stand out slightly from the OLED black background, accompanied by a `#262522` border (thickness `0.8`).

## 2. Perfect Pill Geometry (Curves)
- **Border Radii**: There are no "semi-rounded" rectangles in the top navigation elements. Everything is either a perfect pill or a circle.
  - Date / Month / Week selector pills: `borderRadius: 20` (minimum height 32-40).
  - Navigation arrow buttons (e.g. `ChevronLeft`, `ChevronRight`): `width: 44, height: 44, borderRadius: 22`.
  - Filter category chips/pills: `borderRadius: 20`.
- Do NOT use `borderRadius: 10` or `12` for top-header filter pills. Always opt for the fully rounded `20` curve.

## 3. Huge, Ultra-Bold Typography
- Numerical data must be massive, easily readable at a glance, and ultra-bold.
- **Main Hero Numbers (e.g. Total Income, Net Income)**:
  - Value text: `fontSize: 40`, `fontWeight: "800"`, `color: "#ffffff"`, `letterSpacing: -0.5`, `lineHeight: 48`. Always use `adjustsFontSizeToFit` and `numberOfLines={1}`.
  - Currency symbol: Must be visually distinct. Use `fontSize: 24`, `fontWeight: "600"`, and place it beside the value text, top-aligned using `flexDirection: "row"`. Always separate the symbol using `Intl.NumberFormat` parts.
- **Bento Grid / Stat Cards**:
  - Number values: `fontSize: 32`, `fontWeight: "800"`, `color: "#ffffff"`, `letterSpacing: -0.5`, `lineHeight: 38`, `paddingVertical: 2`, `includeFontPadding: false`. Always use `adjustsFontSizeToFit` and `numberOfLines={1}` to prevent vertical/horizontal text clipping.
- **Labels**: 
  - Standard small headers: `fontSize: 11` or `12`, `fontWeight: "800"`, text-transform `uppercase`, wide letter spacing (e.g. `0.5` or `1`).

## 4. Seamless Tab Navigation (Layout Structure)
- **Preventing Layout Flicker**: When structuring scrollable screens, do NOT apply safe-area `paddingTop` to the outer `<SafeAreaView>`. 
- **Correct Layout Pattern**: Apply the top inset spacing (e.g., `paddingTop: insets.top + 64`) directly to the `<ScrollView contentContainerStyle={{...}}>`. 
- The `<View>` for the screen header (Date selector + navigation arrows) should live *inside* the `ScrollView` so it floats naturally, rather than being a static sticky block with a hard bottom border.
- **Syntax Integrity**: When refactoring screens to float headers inside `ScrollView` components, explicitly ensure that any newly introduced padding or margin `<View>` wrappers are properly closed before the terminating `</ScrollView>` tag to prevent syntax crashes.

## 5. Chart & Graph Aesthetics
- **Graph Container**: Like all Bento cards, the main container wrapping a chart must use `backgroundColor: "#0d0d0d"`, `borderRadius: 20`, `borderWidth: 0.8`, and `borderColor: "#1f1f1f"`.
- **Bar Tracks**: The background track for a vertical bar chart column must be slim and elegant. Use `width: 14` and `backgroundColor: "#161615"` (with a `borderRadius: 7`). Do NOT use thick `20px` bars or `#0d0d0d` tracks.
- **Overlays / Badges**: High-value dashed-line overlays and their text badges should blend seamlessly into the card background, utilizing `backgroundColor: "#0d0d0d"` to match the container surface.

## 6. Type Definitions & Store Safety
- **DriverProfile Interface**: The `DriverProfile` in `store/useSettingsStore.ts` includes optional properties for appearance customization:
  - `bentoLayout?: string` (e.g., `"balanced"`, `"compact"`)
  - `locale?: { currency?: string; dateFormat?: string; weekStartDay?: number; timeFormat?: "12h" | "24h" }`
  When writing settings logic or importing user profile configurations, do not assume they only contain default country/units; make sure code safely references `profile.bentoLayout` and `profile.locale`.
- **Readonly Array Props**: UI selector components such as `Chips` and `Segmented` can receive `as const` arrays (e.g., `WORK_PRESETS`) as options. Ensure that their props are typed using `readonly { value: T; label: string }[]` rather than mutable arrays to avoid TS4104 compiler errors.

## 8. Common Errors & Fixes

- **React Query `Query data cannot be undefined`**: When using `@tanstack/react-query`, if a query function fetches from the local SQLite database using Drizzle (e.g., getting a single row/limit 1) and no records exist yet, the queryFn returns `undefined` (e.g., `res[0]`). Because TanStack Query does not allow returning `undefined`, you MUST fallback to `null` (e.g., `res[0] ?? null`).
- **Duplicate Declaration Babel/Metro Errors**: If Metro throws a `TypeError: Duplicate declaration` error on a file that was recently modified (especially involving imports), it is almost always a bundler cache issue. The standard fix is to clear the Metro cache and restart the Expo bundler using `npx expo start --clear` or `npm run start -- --clear`.
- **React Native Style Percentages (`left: "12%"` etc.)**: When styling with string percentages dynamically in React Native, the TypeScript compiler may reject standard string values as incompatible with `DimensionValue`. To resolve this compile issue, cast the values using `as any` or cast the fields accordingly (e.g. `{ left: seg.left as any }`).
- **React Native Android Widget Linkage Error**: The package `react-native-android-widget` might not be linked in Expo Go or non-widget development environments. Always wrap imports/invocations (like `requestWidgetUpdate`) in a `try-catch` block.
- **Demo Mode Action Restrictions**: When `isDemoMode` is active, write and mutation operations (such as starting live shift tracking, manually adding, editing, or deleting shifts, expenses, and goals) must be guarded. Trigger a native `Alert` warning detailing that the action is disabled in Demo Mode, and offer a button directing them to Settings (`/settings`) to disable Demo Mode.

## 9. App-Wide Accent Customization
- **Theme Hook**: Do NOT use static/hardcoded `DS.brand`, `DS.brandBorder`, `DS.brandSurface`, or `DS.brandText` tokens for interactive items. Instead, use the hook:
  `const { accentColor, accentColorDim, accentColorMid, accentColorContrast } = usePlatformTheme();`
- **Dynamic Accent Colors**:
  - `accentColor`: The primary brand accent color selected by the user (defaults to White `#ffffff`). Used for buttons, text overlays, active state text/icons, and toggle controls.
  - `accentColorDim`: A translucent dim color suitable for background highlight cards and surfaces.
  - `accentColorMid`: A medium-density color used for borders on selected cards or chips.
  - `accentColorContrast`: High-contrast content color (typically `#000000` or `#ffffff` depending on selected accent luminance) to ensure readability of text rendered on top of `accentColor` backgrounds.
- **Component Styling**: Pass style overrides inline dynamically based on selection states (e.g. `style={[s.chip, isSelected && { borderColor: accentColorMid, backgroundColor: accentColorDim }]}`).
- **Default Presets**: Preset colors are defined under `PRESET_ACCENTS` in Settings, and the default is white. Always ensure contrast ratio checks are handled dynamically using `getContrastColor(color)`.

## 10. Scroll-Driven Navigation (Hide/Show & Bottom Reveal)
- **Hysteresis Threshold**: Do not toggle headers or action bars on minor scroll movements. To prevent visual jitter, enforce a scroll difference deadzone of at least 15px (e.g., `diff > 15` to hide, `diff < -15` to show).
- **Auto-Reveal at Scroll End**: Always reveal the top header and bottom action bar when the user reaches the end of the scroll container. Compare the viewport height, content height, and scroll offset to detect the bottom accurately:
  ```typescript
  const contentHeight = event.nativeEvent.contentSize.height;
  const layoutHeight = event.nativeEvent.layoutMeasurement.height;
  const isNearBottom = currentY + layoutHeight >= contentHeight - 40;
  
  if (currentY <= 0 || isNearBottom) {
    setHeaderVisible(true);
  }
  ```
  *Note: A 40px buffer is used to account for subpixel rounding differences on varying device pixel ratios.*
- **Snappy Easing**: For the hide/show slide transitions, use quick and crisp spring physics (`tension: 90`, `friction: 9`) to make the interface feel highly responsive, rather than slow or muddy.
- **Dynamic Bottom Offset**: Position floating bottom bars (like the Action Island) dynamically relative to the safe area bottom inset to avoid clipping behind software navigation keys (Home, Back, Recents) on Android or home indicators on iOS:
  ```typescript
  bottom: insets.bottom > 0 ? insets.bottom + 8 : 16
  ```
- **Adjust Scroll Padding**: Expand the scroll container's bottom padding dynamically (`paddingBottom: BASE_PADDING + insets.bottom`) to offset the raised action bar.
- **Fullscreen Overlay Auto-Hide**: When opening a fullscreen overlay/tracking page (such as the Shift Console / active timer panel), always set `setHeaderVisible(false)` to slide the global top header (and the platform switcher trigger) off-screen, preventing it from overlaying console details. Restore visibility (`setHeaderVisible(true)`) upon minimizing or closing the overlay.

## 11. Compact Platform Management (Settings)
- **Collapsible Inactive Platforms**: To conserve vertical space in the Settings → Platforms tab, only active platforms should be shown fully expanded. Inactive platforms must be grouped into a secondary collapsible section ("Other Available Platforms") and displayed inside a single card.
- **Dynamic Transition**: When an inactive platform is toggled on, it must instantly transition into the "Active Platforms" list and expand to show its configuration fields (such as hourly rate, mileage rate, and sort priority). Conversely, toggling a platform off collapses it and returns it to the inactive pool.
- **Internal Collapsible Configurations**: Inside the active platforms list, each platform configuration card's internal settings (Hourly rate, Mileage rate, Sort priority) must be collapsible. Clicking anywhere on the platform header row (excluding the Switch toggle) will expand/collapse these fields, displaying a Chevron icon indicator and a helper sub-hint ("Configure settings" / "Hide settings"). Newly activated platforms should expand automatically.

## 12. Region-Specific Mileage Presets
- **Onboarding Region-Aware Selector**: During the onboarding setup (specifically `RegionSelectStep` on step 2), users select their province/state/region. The step features a toggle Switch card: "Apply [Preset Label] preset?".
- **Registry Integration**: Default mileage presets are defined in `src/registry/provinces/index.ts` via `getMileagePresetRate` and `getMileagePresetLabel` helper functions:
  - **Canada (CA)**: CRA preset of $0.70/km.
  - **United States (US)**: California (CA) uses Prop 22 active mileage reimbursement rate ($0.35/mi); other US states use the IRS standard rate ($0.67/mi).
  - **United Kingdom (UK)**: HMRC standard rate of £0.45/mi.
- **Cascading Configuration**: On onboarding completion, if the mileage preset toggle was checked, the selected platforms are automatically saved to `platform_configurations` in Settings pre-configured with this regional preset mileage rate (rather than a hardcoded fallback of 0.62). The Settings screen similarly utilizes this registry helper as its fallback when restoring default platform values.

## 13. Read-Only Profiles with Edit Mode Toggle
- **Default Read-Only View**: To prevent accidental edits, the "You" tab in settings is read-only by default. Inputs, Segmented controls, and Chips are replaced by static typography style representations of the value (e.g. `$500` for weekly goal, `Registered` for HST, `Canada` for Country).
- **Edit Mode Trigger**: A small toggle button ("Edit Profile" / "Done") is located next to the "Profile details" section title. Clicking this button toggles the `isEditingYou` local state, transitioning the read-only text elements back into active, interactive inputs/selection chips.
- **Top Bar Save**: Clicking the primary "Save" button in the screen header persists all changed fields regardless of whether the local edit mode is active or collapsed.
- **Locked Regional Fields**: To maintain configuration consistency and prevent calculations mismatch (tax calculations, distance units, etc.), the following settings are locked after setup:
  - **Country**: Displayed as a static text row in edit mode with a "Country cannot be changed after setup." hint.
  - **Distance Unit**: Displayed as a static text row in edit mode with a "Locked to country metric" hint.
  - **Currency**: Locked on the Appearance/Locale tab with a "Locked to your country currency" hint.
  - *Note: Province/State/Region changes within the same country are still fully editable, permitting regional tax rate adjustments when the user moves within their country.*

## 14. Screen-Level Settings Allocation
- **Goals Screen**: Weekly, Monthly, and Annual goals are configured directly on the Goals page. They are excluded from global Settings.
- **Tax Screen**: Tax withholding percentages and HST registration switches (`hstRegistered`) are managed on the Tax screen. They are excluded from global Settings.
- **Schedule Screen**: The `workSchedulePreset` (Flexible, Weekdays Only, Evenings Only, Weekends Only) is managed on the Schedule screen using a bento preset selection card. It is excluded from global Settings.
- **Settings Screen (Tab: You)**: Restricts itself only to the user's Display Name and Location attributes (Country, Province/State, Distance Unit).

## 15. Tax Region Change Guided Flow & Audit Logging
- **Withholding Rate Lookup Fallback**: The withholding rate must be resolved by checking the regional province preset registry first (`src/registry/tax/withholdingPresets`). If no preset exists for the selected region code, fallback to the country-default withholding rate defined in the country registry (`src/registry/countries/index`).
- **Guided Province Change flow**: When a user selects a different tax region (province or state) on the Tax tab screen, intercept the change with a guided confirmation modal. The modal must clearly compare the old region and its withholding rate with the new region and its new rate, warning the user that past logs remain archived under the previous rate while future calculations from today and onward will apply the new rates.
- **Tax Region History Table**: All region changes must be persisted in the `taxHistory` SQLite table (or `localStorage` under `comma_tax_history` on web) to maintain an audit trail. This history list should be displayed chronologically under the config card on the Tax page.
- **Guided Country Change & Platform Audit flow**: When a user selects a different country on the Settings tab, intercept the change with a confirmation modal. The modal must perform a platform compatibility audit: it checks which of the user's active platform configurations are incompatible with the target country, warns the user that they will be deactivated, and provides a side-by-side comparison of currency, distance unit, and default withholding tax rate changes. On confirmation, the change resolves new defaults for the country and inserts a corresponding audit trail record in the `taxHistory` database.

## 16. Modular Platform Registry Architecture

### File Structure
- `src/registry/platforms/types.ts` — `PlatformDef` interface (the shape of every platform)
- `src/registry/platforms/ca.ts` — Canada platforms (DoorDash, UberEats, Skip, Foodora, Instacart, AmazonFlex)
- `src/registry/platforms/us.ts` — US platforms (DoorDash, UberEats, Lyft, Uber, Instacart, AmazonFlex)
- `src/registry/platforms/uk.ts` — UK platforms (UberEats, Deliveroo, Stuart, AmazonFlex)
- `src/registry/platforms/np.ts` — Nepal platforms (Pathao, PathaoFood, InDriver, Foodmandu, Bhoj)
- `src/registry/platforms/global.ts` — Cross-market platforms (AmazonFlex, Other)
- `src/registry/platforms/index.ts` — Barrel: aggregates all, provides `PLATFORM_REGISTRY`, `getPlatformDef`, `getPlatformsByCountry`, backward-compat `PLATFORMS` shim
- `src/registry/operationalModels/index.ts` — The 6 model definitions that drive UI adaptation
- `src/hooks/usePlatformContext.ts` — Hook that resolves a platform ID into full context

### Adding a New Country
1. Add a `CountryDef` to `src/registry/countries/index.ts` and include it in `COUNTRY_MAP`.
2. Create `src/registry/platforms/<countryCode>.ts` with platform definitions using `PlatformDef`.
3. Import the new array into `src/registry/platforms/index.ts` and add to `ALL_PLATFORM_LISTS`.
4. No UI changes needed — everything adapts via the registry.

### Adding a New Platform
1. Create or edit the country-specific platform file (e.g. `np.ts`).
2. Define a `PlatformDef` with the correct `operationalModel` ID.
3. Add it to the country's `<XX>_PLATFORMS` array.
4. That's it — the market resolver, onboarding wizard, and settings will show it automatically.

### OperationalModel IDs (never hardcode platform-specific UI)
- `"delivery_fixed"` — DoorDash, UberEats, Skip, Foodora, Foodmandu (platform sets price, tip optional)
- `"delivery_negotiated"` — Pathao Food, Talabat (negotiated fare, cash common)
- `"rideshare_metered"` — Lyft, Uber (app meters fare with surge)
- `"rideshare_bidding"` — InDriver, Pathao Ride (driver bids/sets price, cash common)
- `"grocery_batch"` — Instacart (batch shopping, per-item bonuses)
- `"parcel_route"` — Amazon Flex (fixed pay per block/route)

### Consuming the Registry in Components
Use `usePlatformContext(platformId)` from `src/hooks/usePlatformContext.ts`:

const { platform, model, terminology, flags, revenueFields } = usePlatformContext(activePlatform);
// terminology.driverTerm → "Dasher" (DoorDash override)

The old `PLATFORMS` flat map (label/color/textColor) is still exported from `src/registry/platforms.ts` as a shim. All existing code importing it continues to work. New code should import `getPlatformDef()` or `PLATFORM_REGISTRY` for full typed access.

## 17. Profile-Aware Tax & Expense Adaptation (Phase 4)
- **Profile-Aware Expense Categories**: Expense category list resolution is decoupled from direct country checks. `getExpenseCategories(countryOrProfile, customCategories)` maps the country to its configured regulatory `expenseProfile` (`cra`, `irs`, `hmrc`, or `generic`) to select the appropriate tax schedule categories.
- **Dynamic Tax UI Gating**: The Tax screen (`app/tax/index.tsx` and `app/(tabs)/tax/index.tsx`) must dynamically adapt layout cards using country definition flags:
  - **Standard Mileage Deduction**: Rendered only when `countryDef.hasMileageDeduction` is true. The write-off preset rate is dynamically fetched via `getMileagePresetRate` and labeled via `getMileagePresetLabel` based on country and region.
  - **Regional Presets**: Province/state selector chips are hidden entirely if the country has no sub-national tax regions (e.g. Nepal, where `regions.length === 0`).
  - **Government Obligations**: CPP (Canada), SE Tax (US), and HST/GST collected tracking are conditionally rendered based on registry flags (`tax.calcCpp`, `tax.calcSeTax`, and `tax.hstOnboarding`).
  - **Tax Deadlines list**: Dynamically calculated and displayed using `countryDef.taxInstallmentDates` configs, avoiding hardcoded date tables. Format dates on-the-fly relative to the current calendar year.
- **Exclusion of Self-Assessment**: If `countryDef.hasSelfAssessmentTax === false` (e.g., tax-free zones or platforms withholding tax at source like in Nepal), the Tax estimator/center is hidden from the main navigation sidebar drawer and the "More" tab. If accessed directly, it renders a simplified notice explaining that self-assessment estimated payments are not required.
- **Cash Operations**: Cash in/out is tracked within individual shifts and expenses, but the app does not support or log individual standalone cash transactions for each day type.
- **Currency Isolation**: The app maintains single-currency operation (matching the active country setting), with no support for multi-currency transactions.
- **Unified Terminology**: If a country does not use self-employment terminology (e.g., `countryDef.hasContractorEconomy === false`), rename "Net Self-Employment Income" to "Net Business Income" / "Net Earnings" or similar generic descriptors.

## 18. Custom Platform Management (Dynamic Upserts & Emojis)
- **User-Defined Platforms**: Allow users to create custom platforms dynamically on the Settings → Platforms tab. The new platform ID is generated as `custom_platform_${Date.now()}`.
- **Visual Parameters**: Custom platforms support user-defined properties: Custom Name (label), Theme Color (hex), and an Emoji Logo (e.g., 🚲, 🚗, 🛵).
- **Upsert Execution**: The query helper `updateDBPlatform` behaves as an upsert: it checks if the platform row already exists in the database. If not, it inserts it; otherwise, it updates the columns (`isActive`, `hourlyRate`, `mileageRate`, `sortPriority`, `label`, `color`, `logoEmoji`).

  - `usePlatformTheme` resolves brand colors and labels dynamically from DB configurations, reflecting custom colors in active filters and top bar metrics.
