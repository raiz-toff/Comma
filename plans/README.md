  reported.

  model, `app/docs/sync-ux-simplification.md`); only enforcement flaws
  (SEC-01/BUG-03) were reported.

## Direction (for the maintainer to weigh — not ranked against the bugs)

1. **Execute the decided sync-engine rewrite** — per-device full-state files +
   one shared `sync-core` module (`app/docs/sync-simplification-2026-07-12.md`,
   complete with cutover table). This audit independently confirmed its
   premise: the duplication has ALREADY drifted (see DEBT-01 row) and most
   unplanned findings above (SEC-03, PERF-02, BUG-06, BUG-08) die with the old
   engine. ~3,100 lines → ~1,000, one implementation. Effort: L (multi-day),
   staged in the doc's own §5 sequence. Prerequisite: Plan 002's tests (they
   pin the parts the rewrite keeps verbatim). When ready, ask for the plan set
   — it should be 3–4 plans following the doc's steps 1–4.
2. **Decide the Expo-web target's fate.** The mobile codebase carries a third
   sync/storage backend (`Platform.OS === "web"` → localStorage) that the
   standalone PWA supersedes. It is reachable (`npm run web`) but effectively
   untested, and it's where BUG-01/SEC-05 live. If the PWA is the only web
   product: strip the `isWeb` branches from `src/services/sync/*` and
   `googleDrive.ts` (M effort, deletes ~300 lines and a whole class of risk).
   If Expo-web ever ships: SEC-05 (tokens in localStorage) must be fixed first.
3. **"Forget this device" UI** — the rewrite's file-per-device model makes
   stale-device cleanup a natural affordance the current design can't offer
   cleanly (doc §3.3). Cheap once the rewrite lands; not before.
